Daily habit · Guarding the main branch

Branch Policies → Rulesets

The guardrails that stop a bad merge. Same intent — protect main — expressed as policy you can read, review, and version.

📐 What maps cleanly

ADO setting → GitHub equivalent · carries over 1:1
  • Require a minimum number of reviewers → required approvals
  • Build must pass → required status checks
  • Linked work item → required linked issue (via checks)
  • Reset votes on new push → dismiss stale approvals
  • Comment resolution → require conversation resolution

⬆️ The upgrade: Rulesets

GitHub only · new capability ADO didn't have
  • Org-wide, layered rules — apply one policy across many repos at once
  • Evaluate / "dry-run" mode before you enforce
  • Stored as config you can export and review like code
  • Bypass lists that are explicit and auditable
The shift in mindset: in ADO you set policies per branch, per repo, by hand. With Rulesets you write the guardrail once at the org level and it covers every repo that matches — governance stops being copy-paste.
Two-column on purpose: left says "your policies survive", right says "and they scale". The killer line is org-level rulesets — anyone who has hand-configured branch policies across 40 repos will feel that pain instantly. Mention evaluate mode: you can see what a rule would block before turning it on.

Making AI-Assisted Development Repeatable

Copilot is fast — but on its own it writes generic code that doesn't know your stack, your security rules, or your conventions. The real win is knowing which lever to pull for which problem. This is that map.

Repository instructions Path-specific rules Prompt files Custom agents Skills

The mental model

Five layers, five jobs

Customization isn't one feature — it's a stack. Each layer answers a different question. The single most common mistake is cramming everything into layer one.

Layer 2 · Path-specific instructions

Rules for certain files only

File*.instructions.md

Some rules belong to just some files — like your tests. One line names which files a rule covers, and Copilot follows it there and nowhere else.

⚠️

One global rule

A test-only rule in the global file leaks into your normal code too.

→
📝

A path-scoped file

→
✅

Right rule, right files

Test files follow test rules. Everything else is left untouched.

🎯  The result: Each rule lives next to the code it governs — no catch-all file stuffed with “but only for tests…” exceptions.

Layer 3 · Prompt files

Save a routine as one command

File*.prompt.md

A task you do again and again — like adding an API endpoint — written down once, then run like a command.

🔁

Done by hand each time

Everyone adds an endpoint a slightly different way, and a step gets missed.

→
📝

A saved command

new-endpoint.prompt.md

→
⚡

One command, every time

Route, schema, test, and docs — built the same correct way for everyone.

🎯  The result: The right way to do a task is written down once and runs identically for the senior and the new joiner.
Layer 3: the highest-leverage layer for teams — one command, the same correct files every time. It freezes the “right way” so it can’t drift.
Layer 4 · Custom agents

A focused mode with limited tools

File.github/agents/*.agent.md

A named mode for one kind of work where you choose which tools it can use — so it physically can’t do the rest.

🤞

You trust a paragraph of instructions to stop the AI touching what it shouldn’t.

→
📝

A tool-limited mode

→
🛡️

It simply can’t

A reviewer with no edit tool cannot change your code — even if you ask it to.

🎯  The result: Safety by capability, not by request — a tool it doesn’t have beats a rule asking it to behave.
Layer 4: skip the “cute persona” framing. The real point is safety by capability — a tool it doesn’t have beats a rule asking it to behave.
Layer 5 · Skills

Knowledge Copilot picks up on its own

FileSKILL.md

A folder of know-how with a short note about when it’s useful. You don’t call it — Copilot reads the note, sees your task matches, and uses it on its own.

📚

Knowledge nobody loads

Expertise sits in someone’s head or a wiki, so Copilot never applies it.

→
📝

A skill with a trigger

SKILL.md
# the description is the trigger description: Use when drafting release notes from merged pull requests.
→
🤖

Copilot pulls it in

You say “draft release notes” and Copilot loads the skill by itself.

🎯  The result: The expert’s know-how fires automatically — no one has to remember the skill exists.
Layer 5: the newest layer. The difference from a prompt file is who starts it — you, or Copilot. The description decides whether it ever fires.
The distinction people miss

Prompt file or skill?

Same question, asked precisely: who pulls the trigger?

🧑 You trigger it → Prompt file

  • You pick it from a list and run it
  • Best for a chore you know you're about to do
  • Always-on intent lives with the human

🤖 The AI triggers it → Skill

  • Copilot recognizes the moment from the description
  • Knowledge is occasionally relevant, not always-on
  • Needs bundled assets — a helper script, a strict template
Rule of thumb: if a human has to remember to trigger it, it's a prompt file. If you want the AI to recognize the moment and pull it in, it's a skill. Write the description like a trigger, not a label.
The disambiguation slide: this is the question developers actually ask. Keep the answer to one axis — who pulls the trigger — and the rest falls out. Reinforce: a vague description means the skill never fires.

Make it last

Treat your customization files like code

Your instructions, prompts, agents and skills change what the AI ships on every request — they’re code, not docs. Left unowned they go stale, and a stale rule is worse than none. So give them what your code already gets:

✓ An owner
Add the paths to CODEOWNERS. Someone is accountable — or it drifts.
✓ PR review
Changing a rule changes the AI’s behaviour. Review it like any code change.
✓ Version control
They live beside the code they govern — they branch, diff and roll back with it.
✓ Tests
Keep a few sample prompts. Re-run them after edits to prove the rule still works.
✓ Clean-up
Every quarter, delete rules pointing at conventions you’ve already dropped.
✓ One source of truth
These files are your coding standard, made executable. Keep them in sync.
The difference that matters: do this and customization compounds. Skip it and you’ve “made some AI docs once” — the thing everyone forgets in a quarter.
Reframe the whole hour: the five layers were the what; this is the so it lasts. Push hard on “a rotten rule is worse than no rule.” This is the slide that separates teams who adopt from teams who dabble. Tie each row back to something they already do for code.
What quietly wastes the effort

The five mistakes

Customization compounds — but only if you avoid the failure modes that make the AI ignore your work.

1 Everything in one file
A 500-line copilot-instructions.md dilutes every rule. Split by scope into layer-2 files.
2 Writing essays, not rules
"We generally prefer, where appropriate…" gets ignored. Write imperatives: "Use X. Never Y."
3 Contradictory instructions
Repo says "prefer async/await", a path file says "use .then() chains" → coin-flip behavior. Audit for conflicts.
4 Set it and forget it
Conventions change; the files don’t follow. Stale rules quietly point Copilot at dead patterns. Own them, review in PRs, prune on a cadence.
5 Vague skill descriptions
"Helps with docs" never triggers; "Use when drafting release notes or a changelog from merged PRs" does. The description is the trigger, not a label.
Close on the failure modes: these are the reasons customization "doesn't work" for teams that tried it. Each maps to an earlier layer. Land the meta-point: a lean constitution plus two good prompt files beats a sprawling instructions doc every time — the AI, like a new teammate, follows clear rules better than long ones.

Who’s accountable

AI can write it and review it — but it can’t approve it

Copilot can author a change and give it a first review. The one thing it can’t do is sign off the merge — that stays human, by your rule.

✓
✍️
AI can write the code
Copilot’s agent can author a change on its own branch.
✓
🔍
AI can review the code
An automatic first pass on every pull request.
✗
🔒

A human with write access must sign off — required by your ruleset.
That’s a policy you own, not a hope. The merge gate is enforced by your required-review rule — and GitHub’s own guidance is to supplement Copilot with a human review.

Make it review to YOUR standard

Four dials that turn a generic reviewer into yours

A default reviewer flags generic issues. The same customization stack you use to write code also governs how Copilot reviews it.

Custom instructions
The same copilot-instructions.md — plus org-level instructions an owner sets once. Your conventions become the review rubric across every repo.
Review effort: Low / Medium
Low is fast, default, catches common issues. Medium (preview) routes to a higher-reasoning model for security-sensitive and cross-service changes — at higher cost.
MCP servers & agent skills
(Preview) Pull context from issue trackers, docs, incident tooling into the review. GitHub and Playwright MCP are on by default; repo skills get reused when relevant.
Where it runs automatically
Configure auto-review at repo or org level — on PR open, on every new push, or even on drafts. On Business & Enterprise, the org setting makes it a standard, not a habit.

Putting Copilot to Work — Autonomously

Customization made Copilot follow your rules. This part hands it whole tasks and lets it run — beside you in the editor, or unattended in the cloud while you do something else. The skill is knowing how much leash to give, and where the leash attaches.

Agent mode (local) Cloud agent (background) Guardrails Hard limits Measuring it
The arc: setup, this is payoff. Customization only matters because something now acts on it autonomously. Throughline: two kinds of autonomy, the same guardrails underneath, and honest limits.
The mental model

Two kinds of autonomy

Both let Copilot plan and act on its own. The difference is where it runs and whether you're watching. Pick by how much you need to steer in real time.

vs
Synchronous · you watch

Agent mode

in your IDE — VS Code, Visual Studio
  • Makes autonomous edits in your local workspace
  • Plans, edits across files, runs tests, self-corrects — live, in front of you
  • You steer mid-flight; nothing leaves your machine until you commit
  • Best when the task needs your judgement turn by turn
Asynchronous · background

Cloud agent

on GitHub — a GitHub Actions environment
  • Works unattended in an ephemeral cloud environment
  • You assign an issue (or @copilot a PR); it researches, codes, tests, opens a PR
  • Every step is a commit in the logs — transparent and reviewable
  • Best for well-scoped work you'd rather not babysit
When to reach for which: agent mode when the task needs your judgement during the work; cloud agent for the tech-debt and "nice-to-haves" that rot on the backlog — review its PR like any teammate's.
The one distinction that matters: local + synchronous (you in the loop every step) vs cloud + asynchronous (you review the PR at the end). Everything else — guardrails, limits — applies to both. The cloud agent was formerly called the "coding agent"; the name changed, the behaviour didn't.
Where the leash attaches

Autonomy is only as safe as its guardrails

The same customization you wrote is what keeps an unattended agent on the rails — plus a few controls specific to autonomy.

Custom instructions
Your repo rules and how to build, test and validate — the agent reads them on every task.
Custom agents
Specialised personas with a restricted toolset — a frontend agent, a docs agent, a test agent.
MCP servers
Scope what data and tools it can reach. GitHub & Playwright MCP are on by default; you control the rest.
Hooks
Run your own shell commands at key points — validation, logging, security scanning — during the agent's run.
Branch protection & rulesets
It still obeys your rules. Required reviews stand; it can't merge its own PR. Add Copilot as a bypass actor only where you mean to.
Required human review
The output is a pull request, not a push to main. A human approves before anything ships.
Guardrails: this is the bridge slide. Make the point explicitly — the reason customization wasn't optional is that autonomy without those rules is the scary version. The PR-not-push design is the single most reassuring fact for nervous stakeholders.
